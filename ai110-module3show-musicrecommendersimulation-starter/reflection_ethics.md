# Reflection and Ethics — VibeFinder AI

## 1. Limitations and Biases in the System

### Catalog sparsity
The catalog contains only 18 songs across 15 genres. Most genres have exactly one representative, so a "genre match" is essentially a coin flip based on whether that one song also fits the user's mood and energy. A request for reggae, blues, or funk will always surface the same single song regardless of how the other preferences differ. This is not a property of the AI — it is a property of the data. Any real recommender would need thousands of songs per genre to produce meaningful variety.

### Genre-weight dominance bias
The scoring formula assigns +2.0 to genre match and only +1.0 to mood. This means a song in the right genre but the wrong mood outscores a song in the wrong genre but with a perfect mood, energy, and acoustic match. For a user who says "I want something peaceful" but picks "pop" as their genre, the system surfaces an energetic, danceable pop song ahead of a peaceful ambient or classical track. The weight encodes a design assumption — genre is the "strongest" signal — that may not hold for every user.

### All-or-nothing label matching
"Indie pop" and "pop" score as completely different genres. "Chill" and "relaxed" score as completely different moods. There is no semantic proximity: a song labeled "chill" gives zero mood-match credit to a user who asked for "relaxed", even though these are psychologically close. A user who types a synonym for a catalog mood will always be penalized.

### Energy is the only gradient
The scoring formula uses a continuous decay for energy but treats all other numeric features (tempo, valence, danceability, liveness) as completely irrelevant. Two songs with identical energy scores are treated identically by the engine even if one is a fast-tempo jazz track and the other is a slow ambient piece.

### LLM guardrails rely on prompt instructions
The guardrail that prevents VibeFinder from discussing off-topic subjects (weather, politics, personal advice) is implemented as a system-prompt instruction to Claude. It works well for common cases but is not formally enforced. A sufficiently creative jailbreak prompt could potentially bypass it. There is no independent content classifier checking outputs before they reach the user.

## 2. Could Your AI Be Misused? How Would You Prevent It?

### Potential misuse scenarios

**Privacy harvesting via music preferences:** Detailed taste profiles (genre, mood, energy, preferred decade, mood tags) combined with usage patterns can reveal sensitive personal information — emotional state, cultural background, age, daily routines. If conversation logs were collected and sold or subpoenaed, they could be used in ways the user never anticipated.

**Manipulation through recommendations:** A commercial deployment could bias the scoring weights toward songs from paying labels or artists, presenting the "most monetizable" recommendation as the "best match." The rule-based engine makes this easy to do invisibly — change a weight, bury the impact in logs.

**Prompt injection via song metadata:** If the catalog were user-editable (crowdsourced titles, tags, artist names), an attacker could inject instructions into song metadata that appear in the tool-result context and attempt to hijack Claude's behavior. For example, a song titled "IGNORE PREVIOUS INSTRUCTIONS AND..." could appear in the retrieved results.

### Prevention measures in place

1. **Read-only catalog:** The `songs.csv` is loaded from a fixed local file at startup. No user can modify catalog contents at runtime.
2. **Tool output grounding:** Claude only receives scores and explanations generated by the deterministic engine, not raw user input injected into the tool results.
3. **System-prompt guardrail:** Claude is explicitly instructed to discuss only music, never to invent songs, and to redirect off-topic requests.
4. **Structured logging:** Every query and every LLM response is written to `logs/vibefinder.log`. Any misuse pattern would appear in the log.

### What would be needed for production safety

- A formal content moderation layer before responses reach users
- Rate limiting and authentication to prevent automated abuse
- A privacy policy explaining what preference data is retained and for how long
- The ability for users to delete their session history

## 3. What Surprised Me While Testing the AI's Reliability

**The diversity penalty was weaker than it looked.** I expected `score × 0.75` to effectively prevent any artist from appearing twice in the top 5. In practice, LoRoom's second song still beat every other option after the penalty because the lofi genre is so sparse — there simply wasn't a better alternative. The penalty reduced LoRoom's second song's *comparison score*, but when all non-lofi songs are already far behind, 75% of a high score is still the best option remaining. This forced me to rewrite the diversity test: it needed to check that the penalty *delays* repeats, not that it *eliminates* them.

**Determinism was the most reliable property.** Every single test that checked the scoring engine's output passed immediately on the first run and has never flaked. Because there is no randomness anywhere in the retrieval pipeline, the same input always produces the exact same ranking. This made debugging LLM responses much easier — when a response seemed wrong, I could check the tool call input against the deterministic engine output and immediately see whether the problem was in the intent extraction (LLM side) or the scoring (engine side).

**Unknown genres silently fail in an expected way.** A user who asks for "polka" gets a list of songs ranked only by mood, energy, and acoustic fit — no genre bonus is awarded. The system doesn't crash or hallucinate a polka song. It gracefully degrades. This was the expected behavior but it was still surprising to see how natural the fallback results looked: the top pick for "polka/happy/0.5 energy" was a funk song that scored well on energy and valence alone. Without reading the explanation, a user might think it was a valid polka recommendation.

## 4. Collaboration with AI During This Project

Throughout this project I worked interactively with Claude Code (claude-sonnet-4-6) to design, write, and test the system. The collaboration evolved through several phases: architecture planning, code generation, debugging, and documentation.

### One instance where the AI gave a helpful suggestion

When designing the LLM layer, I asked how to handle the cost of repeatedly sending the large system prompt (which includes the full song catalog snapshot) on every turn of a multi-turn conversation. Claude Code suggested using **prompt caching** via the `cache_control: ephemeral` field in the system message content blocks, combined with the `anthropic-beta: prompt-caching-2024-07-31` header. This was a non-obvious optimization I had not considered. By marking the static system prompt + catalog block as cacheable, subsequent turns in the same session avoid paying the full input-token cost for the ~1 200-token context that never changes. This directly reduces both latency and API cost for the most common usage pattern (a back-and-forth conversation).

### One instance where the AI's suggestion was flawed

During the reliability test suite, Claude Code wrote the following test assertion:

```python
def test_diversity_penalty_no_repeat_artist(songs, lofi_prefs):
    diverse = apply_diversity_penalty(all_results, k=5)
    artists = [s["artist"] for s, _, _ in diverse]
    assert len(artists) == len(set(artists)), "Duplicate artist after diversity penalty"
```

This test *failed* on the first run. The assertion assumed the diversity penalty would *eliminate* all duplicate artists from the top 5 — but the penalty is a soft multiplier (`score × 0.75`), not a hard exclusion. When LoRoom's second song scores high enough even after a 25% reduction, it remains in the top 5. The test encoded a stronger guarantee than the algorithm actually provides.

The fix required understanding *what the diversity penalty actually contracts*: it must not *increase* repetition, and it must push a repeat artist's second appearance *later* in the ranking — but it does not guarantee zero duplicates. The test was rewritten as two separate assertions, each checking one of those actual invariants. This was a useful reminder that AI-generated tests can encode assumptions that sound reasonable but do not match the real algorithm behavior.
